## 执行阶段说明

当前共分为四个执行阶段（Stage），每次调用只能执行一个阶段：

- **Stage 1**：项目基本信息（base）
- **Stage 2**：技术参数（technical_parameters）
- **Stage 3**：商务条款（business_terms）

## Stage 2：技术参数（technical_parameters）

### 核心原则：语义理解优先

**重要**：不同行业、不同项目的技术参数差异巨大，因此：
- ✅ 依靠语义理解，而非关键词匹配
- ✅ 自动识别技术性质的内容
- ✅ 宽泛提取，允许灵活分类

### 什么是"技术参数"？（判断标准）

满足以下任一条件即为技术参数：

1. **描述产品/设备/系统的物理/性能特征**
   - 尺寸、重量、容量、速度、功率、频率、精度、效率
   - 材质、颜色、形状、结构、成分
   - 处理能力、响应时间、并发数、带宽、存储

2. **规定技术标准或规范要求**
   - 符合GB/T、ISO等标准
   - 达到国家/行业/地方标准

3. **描述制造/施工/安装/调试的工艺方法**
   - 焊接工艺、涂装方法、加工精度
   - 安装步骤、调试流程、检验方法

4. **定义系统配置、组成、接口**
   - 硬件配置、软件版本、网络拓扑
   - 系统架构、模块组成、接口协议

5. **要求测试、验收、试运行**
   - 出厂检验项目、现场测试方法
   - 验收标准、试运行要求

6. **配套设施、备件、工具、文档、培训**
   - 备品备件清单、专用工具、仪器仪表
   - 技术手册、操作手册、培训要求

### 技术性质识别（6大类）

根据内容性质快速判断：

| 类别 | 特征 | 示例 |
|------|------|------|
| **硬件/设备** | 物理设备、机械装置 | CPU、电机、泵、传感器 |
| **软件/系统** | 程序、平台、算法 | 操作系统、数据库、应用软件 |
| **材料/工艺** | 原材料、加工方法 | 钢材规格、焊接工艺 |
| **标准/规范** | 引用标准、规范条文 | GB、ISO、行业标准 |
| **测试/验收** | 检验、测试、验收 | 出厂检验、性能测试 |
| **配套/服务** | 备件、工具、培训 | 备品备件、技术培训 |

### 动态分类系统（8个核心分类）

优先使用以下分类，无法归入时可自定义：

1. **硬件参数**：物理设备、机械设备
2. **软件参数**：软件、系统、平台
3. **性能参数**：速度、精度、效率、容量
4. **环境参数**：温度、湿度、防护等级
5. **标准规范**：引用的标准、规范
6. **工艺要求**：制造、安装、调试方法
7. **测试验收**：检验、测试、验收标准
8. **配套要求**：备件、工具、文档、培训

**允许自定义分类**：如遇到特殊内容，可创建新分类名称。

### 输出结构（JSON）
```json
{
  "technical_parameters": [
    {
      "name": "参数名称/要求标题",
      "value": "参数值/要求描述（尽量完整，包含限定条件）",
      "category": "分类（优先使用上述8个核心分类，无法归入时自定义）",
      "unit": "单位（如有）",
      "remark": "备注（如有）",
      "parameters": [
        {"name": "子参数名", "value": "值", "unit": "单位", "remark": "备注"}
      ],
      "evidence_chunk_ids": ["CHUNK_xxx"]
    }
  ]
}
```

### 抽取原则
1. **宁可多提，不要遗漏**：不确定时优先提取
2. **完整性优先**：保留限定条件（如"≥55kW"、"不低于"）
3. **灵活结构化**：parameters子数组允许为空
4. **动态分类**：category不确定时可用"其他技术要求"或自定义

### 跨行业简要示例

**IT类**：服务器CPU≥2.5GHz、内存≥32GB → 硬件参数  
**工程类**：钢结构满足BS5135标准 → 标准规范  
**设备类**：电机功率≥55kW需配软启动器 → 硬件参数  
**服务类**：提供5×8小时技术支持 → 配套要求

## Stage 3：商务条款（business_terms）

### 职责

**强调**：
- 不做简单关键词匹配
- 同一条款可拆分为多条输出
- 宁可多条，不要合并过度

### 必须覆盖的条款类型
1. 付款方式 / 付款比例 / 节点
2. 交货期 / 服务期 / 履约周期
3. 质保期 / 验收方式
4. 投标资格 / 投标限制
5. 违约责任 / 风险承担
6. 价格构成 / 报价范围
7. 投标保证金 / 履约保证金
8. 合同签订 / 合同管理
9. 发票税费
10. 知识产权 / 保密要求

### 输出结构（JSON）

{

## Stage 4：评分规则（scoring_criteria）- 完整性优先版

### 🚨 最高优先级：完整性 > 准确性 > 格式

**核心原则**：
1. **宁可多提，不要遗漏** - 任何疑似评分内容都要提取
2. **完整复制原文** - 逐字复制，不要总结、改写、简化
3. **保持评分表的连续性** - 不要跳过任何评分项
4. **评分标准关乎投标成败** - 必须100%忠实原文

### 什么内容必须提取？（检查清单）

请逐项检查招标文件片段，确保以下内容都已提取：

#### ✅ 必提取项目

1. **评标办法名称**
   - 综合评分法、最低评标价法、性价比法等
   - 评标办法的说明和描述（如："本项目采用综合评分法，满分100分"）

2. **评分大类**（通常有3-5个大类，全部提取）
   - 商务评审/商务部分
   - 技术评审/技术部分
   - 价格评审/报价部分
   - 资信评审/资质部分
   - 服务评审/售后服务
   - 其他任何评分类别

3. **所有评分细项**（每个大类下的所有子项，一个不漏）
   - 企业资质、项目业绩、财务状况、人员配置
   - 技术方案、实施方案、质量保证、安全措施
   - 价格得分计算方法、报价合理性
   - 服务承诺、质保期、培训方案、售后响应
   - 所有在"评分标准"章节中出现的项目

4. **每项的分值**（原样复制，不要改动）
   - 明确的分值（如：10分）
   - 分值区间（如：5-10分）
   - 最高分值（如：最多10分）
   - 百分比（如：占30%）

5. **每项的得分规则**（这是最关键的！必须逐字复制）
   - 如何得满分的条件
   - 如何得部分分的标准
   - 如何扣分的情况
   - 不得分的条件
   - 计分公式（如果有，必须完整复制）
   - 评分说明（如："评审专家综合评分"）

#### ⚠️ 常见遗漏点（重点检查）

1. **评分表的表头和总分说明**
   - 如："满分100分"、"评分采用百分制"、"权重分配如下"
   - 必须完整提取

2. **价格分计算公式**（重中之重）
   - 如："价格分 = (评标基准价/投标报价) × 30"
   - 如："评标基准价 = 所有有效投标人的投标报价算术平均值"
   - 公式必须完整复制，不得遗漏任何符号和说明

3. **多层级评分项**（层层展开，不要合并）
   - 一级：技术评审（50分）
   - 二级：技术方案（30分）、人员配置（20分）
   - 三级：项目经理（10分）、技术人员（10分）
   - 全部提取，保持层级结构

4. **评分说明和前提条件**
   - 如："评审专家根据投标人提供的技术方案进行综合评分"
   - 如："需提供合同复印件及竣工验收证明"
   - 这些也是规则的一部分，必须完整复制到rule字段

5. **表格内容**（逐行提取，不漏任何一行）
   - 如果评分标准是表格形式
   - 按行逐条提取，不要跳过
   - 保留表格中的所有信息

### 输出结构（JSON）

```json
{
  "scoring_criteria": {
    "evaluationMethod": "【完整复制】评标办法的名称和说明（如：'本项目采用综合评分法，满分100分'）",
    "items": [
      // ⚠️ 注意：items数组应该包含 10-50 个评分项（视招标书复杂度而定）

      {

        "rule": "【完整逐字复制】得分规则的完整原文，包括所有条件、公式、说明、限定词，不得有任何改写、概括或简化",
        "evidence_chunk_ids": ["CHUNK_xxx"]
      }
    ]
  }
}
```

### 质量自检（输出JSON前必做！）

⚠️ **严重警告**：如果评分标准提取不完整（如原文7条只提取6条），将导致投标文件不合规，可能被废标！

在输出JSON之前，请务必进行以下检查：

1. **数量检查**
   - ❓ items数组是否有 **10个以上** 的评分项？
   - ❌ 如果少于10个，说明严重遗漏，请重新仔细检查招标文件片段
   - ✅ 一般的招标项目评分标准应该有15-30个评分项
   - ⚠️ **如果原文有N条评分项，必须全部提取，一条都不能少！**

2. **分值检查**
   - ❓ 所有分值加起来是否接近 **100分**？
   - ❌ 如果总分远小于100（如只有30-50分），说明遗漏了大量评分项
   - ✅ 总分应该在90-100分之间
   - ⚠️ **如果总分明显不足100分，必须回到原文重新检查！**

3. **完整性检查**
   - ❓ 是否包含了 **商务、技术、价格** 三大评分类别？
   - ❓ 每个大类是否都有 **具体的评分项**（不是只有大类标题）？
   - ❌ 如果某个大类下没有具体评分项，说明提取不完整
   - ⚠️ **逐条核对原文，确保每一条评分项都已提取！**

4. **原文检查**
   - ❓ rule字段是否是 **逐字复制** 的原文？
   - ❓ 是否包含了 **计算公式**（如有）？
   - ❓ 是否包含了 **限定条件**（如"近三年"、"提供证明材料"等）？
   - ❌ 如果有改写、概括、简化，必须重新提取

### 典型错误示例（严禁出现）

❌ **致命错误1：遗漏大量评分项**
```json
{
  "items": [
    {"category": "商务评审", "item": "企业资质", "score": "10分", "rule": "..."},
    {"category": "技术评审", "item": "技术方案", "score": "30分", "rule": "..."},
    {"category": "价格评审", "item": "报价", "score": "30分", "rule": "..."}
  ]
}
// ❌ 只有3项？招标文件通常有15-30个评分项！
// ❌ 总分只有70分？还有30分去哪了？
// 这是严重错误！必须重新提取！
```

❌ **致命错误2：改写原文**
```json
{
  "rule": "有相关资质得分，无资质不得分"
}
// ❌ 原文是："投标人具有建筑工程施工总承包壹级及以上资质的得10分，
//          贰级资质的得6分，叁级资质的得3分，无资质不得分。"
// 你改写和简化了！这是严重错误！
// 必须逐字复制原文！
```

❌ **致命错误3：省略计算公式**
```json
{
  "rule": "根据投标报价计算得分"
}
// ❌ 原文有具体公式："评标基准价 = 所有有效投标人的投标报价算术平均值。
//                     价格分 = (评标基准价/投标报价) × 30"
// 必须完整复制公式，不得省略！
```

❌ **致命错误4：合并评分项**
```json
{
  "item": "体系认证",
  "score": "5分",
  "rule": "有认证得分"
}
// ❌ 原文是两个独立的评分项：
//    "1. 质量管理体系认证（3分）"
//    "2. 环境管理体系认证（2分）"
// 不要合并！应该分为两条提取！
```

   - 不要担心JSON太大
   - 必须完整复制，不得截断或简化
   - 保持原文的所有信息

     {category: "技术评审-技术方案", item: "实施方案", score: "15分", ...}

   - 如果只找到部分评分内容，也要全部提取
   - 不要因为不完整就放弃提取
   - 在evidence_chunk_ids中标注所有相关chunks

5. **评分规则引用了其他文件**
   - 如："详见附件X"、"按照《XX标准》执行"
